# Testing

A feature is not done because it compiles. It is done when it compiles, works in-game, and does not break the project’s core loops.

## Minimum commands

Run from the repository root:

```bash
./gradlew compileJava
```

When resources, datagen, recipes, loot tables, or models are changed:

```bash
./gradlew runData
./gradlew validateResources
./gradlew compileJava
```

Before a release or large merge:

```bash
./gradlew build
```

On Windows, use `gradlew.bat` instead of `./gradlew`.

## Manual test checklist

For every feature PR, test:

- [ ] New items/blocks appear correctly where intended.
- [ ] No missing textures or models.
- [ ] No missing lang keys.
- [ ] Recipes load and appear in JEI if they should.
- [ ] The in-game guide is updated if progression changed.
- [ ] The feature works in survival, not only creative.
- [ ] The feature does not duplicate drops/items.
- [ ] Breaking blocks and unloading/reloading chunks behaves correctly.
- [ ] Client-only visuals do not crash dedicated server.
- [ ] Network payloads are server-authoritative where needed.

## Core smoke test

Use this after larger changes:

1. Create a new world.
2. Obtain early seeds/items.
3. Grow or process at least one early crop.
4. Use the drying rack.
5. Use the grinding bowl.
6. Consume coffee or tobacco.
7. Confirm HUD/effects appear correctly.
8. Open JEI and check relevant categories.
9. Place and break a new block from the PR.
10. Save, quit, reload, and verify state persists.

## Dedicated server safety

Any PR touching renderers, screens, overlays, particles, keybinds, shaders, or client state needs a dedicated-server check.

Watch for:

- common classes importing `net.minecraft.client.*`;
- server events referencing client-only classes;
- payload handlers registered on the wrong side;
- block entities trying to instantiate render state on server.

Audit command:

```bash
rg -n "import net\.minecraft\.client|import net\.neoforged\.neoforge\.client|import org\.mydrugs\.mydrugs\.client" src/main/java/org/mydrugs/mydrugs -g "!**/client/**"
```

## Datagen checks

If a PR changes generated data:

- [ ] Run `runData`.
- [ ] Run `validateResources`.
- [ ] Check generated files for unexpected mass rewrites.
- [ ] Do not hand-edit files that are generated by providers.
- [ ] Keep source content, such as `docs/progression_guide_pages.md`, separate from generated output.

`validateResources` checks:

- JSON files under `src/main/resources` and `src/generated/resources` are UTF-8 without BOM.
- Registered blocks have blockstates.
- Registered items have client item definitions/models.
- Registered item/block lang keys are present in the canonical hand-written `en_us.json`.
- Guide `@item` references resolve to registered items or blocks.
- `src/generated/resources/.cache/` is ignored.
- Backup/checkpoint files such as `*~`, `*.bak`, `*.orig`, `.rej`, `.swp`, and `.swo` are absent.
- Missing texture/model references are recorded in `docs/ASSET_TODO.md`.

Localization audit:

```bash
rg -n "Component\.literal\(" src/main/java/org/mydrugs/mydrugs
```

Network/rate-limit audit:

```bash
rg -n "PayloadRateLimiter|playToServer|registerPayload|ByteBufCodecs\.STRING_UTF8|\.ordinal\(" src/main/java/org/mydrugs/mydrugs
```

`PayloadRateLimiter` currently uses these minimum intervals:

- roller drag, coffee pulper drag, sieve shake: 45 ms;
- stimulant dash: 200 ms;
- Psy Mixer ritual input: 150 ms.

It also removes logout state and periodically purges player entries stale for more than 10 minutes.

## Balance test notes

For effect changes, record:

- duration;
- intensity;
- dose state behavior;
- HUD clarity;
- what makes the effect worth using;
- what the consequence is.
